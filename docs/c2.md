# Distributed SDR via Meshtastic C2

The central node (`sdr.py server`) can optionally coordinate remote sensor
nodes over a Meshtastic mesh. Remotes run `sdr.py agent`, get tasked from the
central node, and stream detections back.

## Central node setup

Add a `meshlink` section to `configs/server.json`:

```json
{
  "meshlink": {
    "port": "/dev/serial/by-id/usb-Meshtastic_RAK4631-...",
    "channel_index": 0
  },
  "captures": [ ... ]
}
```

Then install the service:

```sh
sudo ./scripts/install-server.sh
```

## Remote node setup

1. Flash the Raspberry Pi, clone this repo, create venv, install requirements
2. Connect RTL-SDR, Meshtastic, GPS via USB
3. Configure the Meshtastic device (shared channel PSK across nodes)
4. Run the installer:

```sh
sudo ./scripts/install-agent.sh
```

Provide an agent ID (e.g. `N01`), the Meshtastic USB path, and GPS port.

## Adoption

On first boot, the agent broadcasts `HELLO` every 30 s.
Open the central web UI → **Agents** tab → click **Approve** for each new node.

## Tasking

From the Agents tab:
- **Start...** — pick a scanner (`pmr`, `ism`, `wifi`, ...) and optional args
- **Stop** — stop the current scanner

Rate-limit and interval settings propagate via `CFG` messages.

## Resilience

- Agents retry queued detections with exponential backoff until ACKed
- Agents resume the last-tasked scanner after reboot
- No data loss on mesh outage — the persistent outbox drains when contact resumes

## Wire protocol

See `src/comms/protocol.py` for the full encoder/decoder. Messages are plain
text, pipe-delimited:

| Tag | Direction | Meaning |
|---|---|---|
| `HELLO` | agent → central | adoption beacon |
| `APPROVE` | central → agent | finalise adoption |
| `CMD` | central → agent | start/stop scanner, set param, request status |
| `CFG` | central → agent | update agent/comms config (rate, intervals) |
| `RES` | agent → central | reply to CMD/CFG/APPROVE |
| `DET` | agent → central | detection, with agent_id + seq + lat/lon |
| `STAT` | agent → central | periodic health (scanner, GPS, CPU, uptime) |
| `LOG` | agent → central | occasional diagnostics |
| `ACK` | central → agent | acknowledges DET/STAT/LOG by seq |

## Storage

All mesh detections land in a single `output/agents_YYYYMMDD_HHMMSS.db`
per server session. The existing `detections` table is reused; `device_id`
holds the agent ID. The standard web dashboard and triangulation tools work
on these rows transparently.

Approved-agent registry is stored in `output/agents_state/agents.json`.
